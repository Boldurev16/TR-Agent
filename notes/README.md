# TR Agent

Форк проекта «10 001 PDF»: локальный агент для PDF и данных тех. радара через **AnythingLLM** с локальной LLM (**Ollama** и/или **LM Studio**).

Учебная база — гайд AiManual по автономному исследовательскому агенту.

- Гайд: [Автономный исследовательский агент на AnythingLLM (10 000 PDF, 32 ГБ ОЗУ)](https://ai-manual.ru/article/avtonomnyij-issledovatelskij-agent-na-anythingllm-kak-obrabotat-10-000-pdf-lokalno-na-32-gb-ozu/)

## Структура проекта

- `docs-inbox` — исходные PDF.
- `docs-batches` — копии PDF партиями (`batch-0001`, …).
- `scripts` — проверка окружения и вспомогательные скрипты.
- `config/llm-presets.example.env` — шаблон Base URL / Model id для копирования в AnythingLLM.
- `cache` — временные данные (позже: извлечённый текст).
- `logs` — журналы обработки и ошибок.
- `notes` — этапы и заметки.

## Обучение по шагам (конспект)

Цель этого блока: вы сами повторяете цепочку «кто куда стучится» и учитесь **осознанно** менять модель, а не наугад кликать в UI.

### Урок 1 — диагностика (вы уже сделали)

**Что делали:** писали `scripts/check_environment.ps1` и гоняли его из терминала.

**Зачем:** любая ошибка «не отвечает» сначала сводится к вопросам: сервер LLM жив? AnythingLLM жив? Docker жив? Какой **точный** `id` модели видит API? Без этого переключение моделей превращается в угадайку.

**Повторение:** из корня проекта `.\scripts\check_environment.ps1` — перед каждым новым экспериментом с моделью имеет смысл прогонять снова.

### Урок 2 — одна цепочка запроса (обязательно понять)

1. **Вы** пишете сообщение в веб-интерфейсе AnythingLLM.
2. **AnythingLLM** (часто в **Docker**) решает: какой провайдер LLM, какой Base URL, какая строка **Model**.
3. Запрос уходит на тот адрес. Если провайдер — LM Studio на вашем ПК, то из контейнера адрес **не** `http://127.0.0.1:1234`, а **`http://host.docker.internal:1234`** (плюс путь `/v1`, как требует ваш провайдер в настройках AnythingLLM).
4. **LM Studio** принимает запрос и считает ответ **только на той модели, которая у вас сейчас загружена** в приложении.

**Вывод для «на лету»:** «переключить модель» значит (а) в LM Studio загрузить другую модель и (б) в AnythingLLM указать **тот же** `id`, который выдаёт `GET .../v1/models` для этой модели. Два шага, а не один.

### Урок 3 — ваш личный файл пресетов (сделайте сами, следующий практический шаг)

1. Скопируйте `config/llm-presets.example.env` в `config/my-llm-presets.env` (имя любое, главное — **не коммитить секреты**, у вас их пока нет).
2. Заполните **реальными** значениями после того, как `check_environment.ps1` показал список моделей:
   - `ANYTHINGLLM_LLM_BASE_URL`
   - `ANYTHINGLLM_LLM_MODEL_ID` (например `google/gemma-4-31b`)
3. В AnythingLLM при смене эксперимента открываете **свой** файл и руками переносите два поля в настройки воркспейса.

**Зачем:** вы фиксируете «конфигурацию эксперимента» текстом. Это и есть учебная дисциплина для переключения моделей.

### Урок 4 — честное сравнение двух моделей (A/B)

1. Создайте **два воркспейса** в AnythingLLM, например `AB-gemma-31b` и `AB-qwen-35b`.
2. Одинаковый Base URL, **разный** `ANYTHINGLLM_LLM_MODEL_ID` в вашем `my-llm-presets.env` (две секции или два файла — как удобно).
3. Правило: перед тестом **в LM Studio загружена** та модель, с которой сейчас говорите; иначе ответ может быть неверным или с ошибкой.

**Зачем:** один воркспейс — один «режим»; вы не смешиваете настройки в голове.

### Урок 5 — PDF в AnythingLLM (RAG, не «модель сама читает файл»)

**Важно понять:** чат-модель (Nemotron, Gemma и т.д.) **не открывает PDF с диска**. Она видит только то, что ей передал AnythingLLM. Схема такая: PDF загружают в воркспейс → AnythingLLM **извлекает текст**, режет на фрагменты, строит **векторный индекс** (нужна отдельная **модель эмбеддингов**) → при вопросе подмешиваются релевантные куски в промпт → модель **формулирует ответ по этому контексту**. Это и есть RAG из гайда [AiManual](https://ai-manual.ru/article/avtonomnyij-issledovatelskij-agent-na-anythingllm-kak-obrabotat-10-000-pdf-lokalno-na-32-gb-ozu/).

**Пошагово (ваш воркспейс «TR Agent»):**

1. **Документы в проекте:** положите PDF в `docs-inbox`, из корня проекта выполните `python .\scripts\make_pdf_batches.py`. Берите файлы для загрузки из `docs-batches\batch-0001` (или любой первой пачки) — так вы приучаетесь к конвейеру из гайда.

2. **В AnythingLLM — загрузка:** в этом же воркспейсе найдите раздел **Documents** / **Workspace documents** / **Upload** (название зависит от версии), загрузите **1–3 PDF** для теста. Дождитесь окончания **индексации** (статус «embedded» / без ошибок).

3. **Модель эмбеддингов:** в настройках воркспейса отдельно от «Chat model» есть **Embedding** / **Vector database** / привязка к API. Нужно указать работающий эмбеддер. Варианты: встроенный локальный (если AnythingLLM предлагает), **LM Studio** с моделью вроде `text-embedding-nomic-embed-text-v1.5` (если у вас она есть — тот же хост `host.docker.internal`, порт LM Studio, модель эмбеддингов в настройках AnythingLLM), либо Ollama с embedding-моделью — **смотрите выпадающий список провайдеров** в вашей версии. Без успешных эмбеддингов поиск по документам не работает.

4. **Режим чата:** в диалоге включите использование документов воркспейса, если есть переключатель **Workspace chat** / **Query mode** / attach context (в разных версиях по-разному). Задайте вопрос **конкретно по формулировке из PDF** (цитата, название раздела, уникальная фраза).

5. **Критерий успеха:** в ответе появляются факты из файла; в UI часто видны **источники** / citations. Если модель «рассуждает вообще» без цитат — контекст из PDF не подмешался; вернитесь к шагам 2–4.

**Урок 6 (следом в проекте):** массовое извлечение текста в `cache/` и логирование (как в гайде — `unstructured` и кэш), чтобы не упираться в наивную загрузку десятков тысяч PDF только через UI.

- Скрипт `scripts/extract_pdf_to_cache.py`: один файл (путь или самый новый в `docs-inbox`), либо все PDF из входящих — `python .\scripts\extract_pdf_to_cache.py --all`; при повторных прогонах удобно `--skip-existing`. Лог: `logs/extract-pdf.log`.

## LM Studio: Gemma 4 31B и AnythingLLM

Гайд изначально про Ollama, но тот же AnythingLLM можно кормить **локальным OpenAI-совместимым API** из [LM Studio](https://lmstudio.ai/). Модель [google/gemma-4-31b](https://lmstudio.ai/models/google/gemma-4-31b) (мультимодальная, инструменты, большой контекст) ставится через приложение: скачать веса, открыть вкладку загрузки модели, при необходимости выставить размер контекста и GPU offload.

### Подключение к AnythingLLM (у вас Docker)

На **хосте** Windows LM Studio слушает обычно `http://127.0.0.1:1234`. Внутри контейнера AnythingLLM `127.0.0.1` — это сам контейнер, а не ПК. Укажите базовый URL провайдера LLM так:

- **Base URL:** `http://host.docker.internal:1234/v1`
- **API Key:** произвольная строка (например `lm-studio`), если LM Studio не требует реальный ключ.
- **Модель:** точная строка `id` из списка моделей LM Studio (см. вывод `check_environment.ps1` или вкладка сервера в LM Studio). Обычно это совпадает с именем загруженной модели.

В LM Studio: включите **Local Server**, загрузите **Gemma 4 31B**, затем в AnythingLLM в настройках воркспейса выберите совместимый провайдер (часто «OpenAI Generic» / кастомный endpoint) и вставьте URL выше.

### Менять модели «на лету» — три практичных уровня

1. **Только LM Studio (быстрый A/B тест):** в LM Studio снимите одну модель, загрузите другую, **тот же** Base URL и тот же воркспейс. В настройках AnythingLLM обновите поле **Model** на новый `id` из `/v1/models` (или перезапросите список в UI). Один воркспейс — одна активная модель за раз.

2. **Несколько пресетов без правок Docker:** создайте в AnythingLLM **несколько воркспейсов** («Проект A — Gemma 31B», «Проект B — Nemotron» и т.д.) и в каждом задайте свой **Model id**. Сервер LM Studio в каждый момент времени реально исполняет **одну** загруженную модель: если второй воркспейс запросит другой `id`, LM Studio либо подставит загруженную, либо вернёт ошибку — для честного A/B **переключайте модель в LM Studio** и держите `id` в настройках синхронно.

3. **Два локальных бэкенда:** параллельно держать **Ollama** (порт `11434`) и **LM Studio** (`1234`) и в разных воркспейсах указывать разные Base URL — тогда переключение «между стеками» делается выбором воркспейса, без остановки Docker.

Идея для обучения: завести текстовый файл с двумя-тремя строками конфигурации (Base URL + Model id + заметка), копировать в UI при эксперименте — так вы воспроизводите «конфигурацию» без угадывания.

## Команды

Из корня проекта `c:\AI Agent\TR Agent`:

```powershell
.\scripts\check_environment.ps1
```

```powershell
python .\scripts\make_pdf_batches.py
```

Извлечение текста в `cache/` (урок 6): один PDF или все из `docs-inbox`:

```powershell
python .\scripts\extract_pdf_to_cache.py
python .\scripts\extract_pdf_to_cache.py --all
```

Создание кастомной модели Ollama (после `ollama pull llama3.2:13b-q4_K_M`):

```powershell
ollama create my-researcher -f .\config\Modelfile
```
Если образ с тегом `llama3.2:13b-q4_K_M` в вашей версии Ollama называется иначе, откройте `config/Modelfile` и поправьте строку `FROM ...` под результат `ollama list`.
